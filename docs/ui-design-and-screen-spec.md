# 온라인 투표 서비스 UI Design Direction & Screen Specification

## 1. 전체 UI 디자인 원칙

본 문서는 관리자 포털과 투표자 포털의 UI 방향, 화면 구성, 공통 컴포넌트, 상태 표현, MVP 화면 범위를 정의한다. 구현 코드, API, ERD, 와이어프레임은 포함하지 않는다.

| 원칙 | UI 적용 방향 |
| --- | --- |
| 신뢰감 | 투표 상태, 일정, 권한, 결과 확정 여부를 항상 명확히 표시한다. 장식보다 정보의 정확성과 안정감을 우선한다. |
| 공공성 | 공문서에 가까운 중립적 문구와 절제된 시각 스타일을 사용한다. 과도한 마케팅 톤, 감정적 문구, 불필요한 애니메이션을 피한다. |
| 단순성 | 투표자 흐름은 단계와 입력을 최소화한다. 관리자 화면은 복잡한 정보를 카드, 표, 체크리스트로 나누되 한 화면의 의사결정 단위가 명확해야 한다. |
| 관리자 실수 방지 | 시작, 중단, 종료, 확정, 공개, 무효, 권한 변경 등 위험 작업은 상태 제한, 확인창, 사유 입력, step-up 인증, 감사 안내를 동반한다. |
| 투표자 접근성 | 큰 터치 영역, 쉬운 문장, 명확한 오류 메시지, 모바일 우선 레이아웃을 사용한다. 색상만으로 상태를 구분하지 않는다. |
| 모바일 사용성 | 투표자 포털은 360px 폭에서도 주요 CTA가 보이고, 문항 선택과 제출 확인이 한 손 조작에 적합해야 한다. |
| 개인정보 보호 | 개인정보는 기본 마스킹한다. 관리자 화면에서도 역할과 업무 목적에 필요한 최소 정보만 노출한다. |
| 익명투표 신뢰성 | 익명투표에서는 투표 전 안내, 제출 후 제한된 receipt, 관리자 노출 제한을 UI에 일관되게 반영한다. |
| 위험 작업 경고 | 되돌릴 수 없거나 결과 신뢰성에 영향을 주는 작업은 주황/빨강 계열 경고, 명확한 사유 입력, 감사 기록 안내를 제공한다. |
| 상태 변화 명확성 | Election 상태 변화는 StatusBadge, 페이지 헤더, 상태 타임라인, CTA 제한으로 동시에 표현한다. |

## 2. 디자인 톤앤매너

| 항목 | 추천 방향 |
| --- | --- |
| 전체 분위기 | 조용하고 신뢰감 있는 SaaS Admin Dashboard. 투표자 화면은 공공 서비스처럼 단순하고 안정적인 모바일 우선 경험. |
| 컬러 방향 | 흰색/회색 기반, 파란색 또는 남색 계열 포인트. 위험은 주황/빨강, 성공은 초록, 대기/검수는 파랑 또는 보라가 아닌 중립 계열을 사용한다. |
| 타이포그래피 | 본문 가독성을 우선한다. 관리자 표/폼은 14px 내외, 투표자 주요 CTA와 문항 제목은 더 크게 표시한다. 과도한 굵기와 좁은 자간을 피한다. |
| 여백/간격 | 관리자 화면은 정보 밀도를 유지하되 섹션 간격을 명확히 둔다. 투표자 화면은 문항, 선택지, CTA 사이 간격을 넉넉히 둔다. |
| 버튼 스타일 | Primary는 단일 핵심 CTA에만 사용한다. 위험 작업은 danger 버튼과 경고 확인창을 함께 사용한다. 비활성 버튼에는 이유를 툴팁 또는 설명으로 제공한다. |
| 카드 스타일 | 카드 반경은 8px 이하. 반복 정보, 요약 지표, 안내문에 사용한다. 페이지 전체를 카드 중첩 구조로 만들지 않는다. |
| 테이블 스타일 | 관리자 목록/명부/로그에 사용한다. 고정 헤더, 필터, 상태 배지, 마스킹 표시를 지원한다. 익명투표에서는 개인별 제출 상세 필드가 테이블에 나타나면 안 된다. |
| 폼 스타일 | 단계형, 섹션별 저장, 필수값 표시, 시작 후 수정 불가 안내를 제공한다. 위험 설정은 별도 확인을 요구한다. |
| 모달/확인창 | 단순 확인, 사유 입력, step-up 인증, 이중 승인 요청을 목적별로 분리한다. 모달 안에 복잡한 설정 편집을 넣지 않는다. |
| 위험 작업 경고 | 작업 영향, 되돌릴 수 있는지 여부, 감사 기록 여부, 필요한 승인자를 한눈에 보여준다. |
| 상태 표시 | 성공은 완료/확정, 실패는 오류/반려, 대기는 검수/집계, 중단은 Paused, 무효는 Invalidated로 명확히 구분한다. |
| 익명투표 안내 | 익명투표 전용 안내 카드를 사용한다. “관리자도 선택 내용을 볼 수 없습니다”와 “이전 선택 내용은 다시 표시되지 않습니다”를 분리해 설명한다. |

## 3. 관리자 포털 화면 구조

관리자 포털은 좌측 내비게이션, 상단 조직/역할/알림 영역, 본문 업무 화면의 3영역 구조를 기본으로 한다. 상태와 권한에 따라 CTA는 숨김 또는 비활성 처리하고, 비활성 사유를 표시한다.

| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |

## 4. 투표자 포털 화면 구조

투표자 포털은 모바일 우선 단일 흐름을 기본으로 한다. 투표자는 관리자 기능, 감사 로그, 개인별 내부 식별자, 이전 익명투표 선택 내용을 볼 수 없다.

## 7. AuthenticationPolicy UI 표현

| `external_identity` | 외부 본인확인 | 기본 비활성 | 높은 보안이 필요한 투표에서 사용합니다. | 제공자, 인증 결과 정책 | 유료/확장 옵션 |
| `sso` | SSO | 기본 비활성 | 조직 계정과 연동합니다. | SAML/OIDC 제공자, 도메인 제한 | 연동 옵션 |
| `legal_strong_auth` | 법적 효력 모드용 강한 인증 | MVP 제외 | 법적 효력 투표 전용 별도 모드입니다. | 본인확인, 전자서명, 증빙 보관 | 향후 별도 설계 |

UI 규칙:

- 인증 방식 선택은 라디오 카드 또는 segmented control이 아니라 설명형 선택 카드로 제공한다. 보안 수준, 비용 여부, MVP 사용 가능 여부를 함께 표시해야 한다.
- 코드 기반 방식이 선택되지 않은 경우 인증코드 재발송, 인증코드 만료, 인증코드 실패 잠금, 코드 발송 채널 설정은 숨긴다.
- 비용 발생 방식은 "기본 비활성", "비용 발생 가능", "상용화 옵션" 배지를 표시한다.
- 조직에서 허용하지 않은 방식은 비활성화하고 이유를 표시한다.
- 법적 효력 모드는 MVP에서 선택할 수 없으며 "향후 별도 모드"로만 표시한다.

## 8. 익명투표 UI 주의사항

익명투표 UI는 데이터 모델의 분리 원칙을 화면에서도 유지해야 한다.

| 대상 | UI 원칙 |
| --- | --- |
| 관리자 참여 현황 | 기본은 집계 단위로 제공한다. 유권자별 상태가 필요한 경우 "미참여 / 참여 완료 / 인증 문제 있음 / 초대 발송 실패" 정도로 제한한다. |
| 관리자에게 금지되는 정보 | 개인별 Ballot ID, Vote ID, AnonymousBallotGroup ID, ballot_group_token_hash, 개인별 상세 제출 시각, IP, User-Agent를 표시하지 않는다. |
| 로그 조회 | CredentialEvent와 SubmissionEvent를 개인 단위로 연결해 보여주지 않는다. 인증 시각과 제출 시각을 같은 유권자 행에 나란히 표시하지 않는다. |
| 투표자 완료 확인 | 완료 여부, 마지막 제출 시각, 제한된 receipt 일부만 표시한다. 선택 내용과 완료 상태를 분리한다. |
| 재투표 | 이전 선택 내용을 다시 보여주지 않는다. 빈 투표 입력 화면에서 다시 시작한다. |
| 결과 화면 | 집계 결과만 표시한다. 소규모 익명투표는 재식별 위험이 있으면 상세 결과를 제한한다. |
| 안내 문구 | 투표 전 익명성 보장 범위와 남는 한계를 필수로 표시한다. |

추천 안내 문구:

- "이 투표는 익명투표입니다. 관리자도 특정 유권자의 선택 내용을 볼 수 없습니다."
- "투표 완료 여부와 투표 선택 내용은 분리되어 관리됩니다."
- "마감 전 다시 투표할 수 있지만, 익명투표에서는 이전 선택 내용을 다시 표시하지 않습니다."
- "소규모 투표에서는 결과 공개 범위가 제한될 수 있습니다."

## 11. 구현 시 UI 라이브러리 고려사항

현재 프로젝트 구조를 확인한 결과, 프론트엔드 프레임워크나 UI 라이브러리 설정 파일은 아직 확인되지 않았다. 따라서 설치나 구현은 하지 않고, 다음 기준으로 후보만 제안한다.

| 기준 | 고려사항 |
| --- | --- |
| 빠른 구현 가능성 | 관리자 대시보드, 표, 폼, 모달을 빠르게 만들 수 있어야 한다. |
| 접근성 | Dialog, Select, Tabs, Tooltip, Focus trap 등 기본 접근성이 확보되어야 한다. |
| 테이블/모달/폼 지원 | 명부, 로그, 권한, 결과 검수에 DataTable과 Form validation이 중요하다. |
| Tailwind 호환성 | 일관된 디자인 토큰과 반응형 구현이 쉬워야 한다. |
| 관리자 대시보드 | 필터, 배지, 카드, 상태 타임라인, 위험 확인창 구성이 쉬워야 한다. |
| 모바일 투표자 화면 | 큰 CTA, 단계형 흐름, 오류 상태, 제출 중 상태를 안정적으로 표현해야 한다. |

추천 후보:

- React 기반으로 구현한다면 `shadcn/ui` + `Radix UI` + `Tailwind CSS` 조합을 우선 검토한다.
- 테이블은 `TanStack Table`, 폼은 `React Hook Form` + `Zod` 조합을 검토한다.
- 다른 프레임워크를 선택한다면 동일 기준의 접근성 중심 headless component와 검증 가능한 form/table 라이브러리를 우선한다.
- 보안상 중요한 ConfirmDialog, ReasonInputDialog, SensitiveInfoMask, AuditNotice는 서비스 전용 컴포넌트로 표준화한다.

### 구현 시 주의할 UI/보안 사항

- 익명투표에서 관리자에게 개인별 Ballot ID, Vote ID, AnonymousBallotGroup ID, ballot_group_token_hash, 개인별 상세 제출 시각, IP, User-Agent를 노출하지 않는다.
- CredentialEvent와 SubmissionEvent를 개인 단위로 연결해 보여주는 화면을 만들지 않는다.
- 투표자에게 익명투표의 이전 선택 내용을 다시 보여주지 않는다.
- 코드 인증 설정은 코드 방식이 활성화된 Election에서만 표시한다.
- Published 이후 결과는 덮어쓰기 UI를 제공하지 않고 정정 요청, 새 ResultVersion, 정정 공지, 무효 공지 흐름으로만 제공한다.
- 위험 작업은 확인, 사유 입력, step-up 인증, 이중 승인 필요 여부, AuditEvent 기록 안내를 표준화한다.
- 개인정보 원문 보기, 보고서 다운로드, 감사 로그 다운로드는 목적 입력과 감사 기록을 요구한다.

### API 설계 전에 추가 확인할 사항

- 화면 권한별 필드 노출 정책을 API 응답 스키마와 어떻게 분리할지 확정해야 한다.
- 익명투표 참여 현황 API가 개인별 상세 제출 시각을 반환하지 않도록 응답 범위를 제한해야 한다.
- AuthenticationPolicy별 설정 필드가 조건부로 반환/검증되도록 정책을 정의해야 한다.
- 위험 작업 API에는 사유, step-up 인증 결과, 승인자, 이전/이후 상태가 포함되어야 한다.
- 보고서/로그 다운로드 API는 목적 입력, 승인 상태, 워터마크 식별자를 포함해야 한다.
- 투표자 완료 확인 API는 선택 내용이 아니라 완료 여부, 마지막 제출 시각, 제한된 receipt만 반환해야 한다.
