# 온라인 투표 서비스 앱 설계서

## 1. 서비스 개요

### 서비스 목적

본 서비스는 폐쇄형 유권자 명부를 기반으로 조직과 단체가 온라인 투표를 안전하게 개설, 운영, 확정, 공개할 수 있도록 지원하는 Web Voting SaaS이다. 핵심 목표는 빠른 투표 생성보다 투표 신뢰성, 개인정보 보호, 강한 익명성, 운영 감사 가능성을 보장하는 것이다.

### 서비스 대상

- 협회, 조합, 사내 조직, 위원회, 비영리 단체 등 폐쇄형 유권자 명부를 가진 조직
- 조직 내부 의사결정용 투표, 대표 선출, 중요 안건 찬반, 안건 선택, 의견 수렴
- 기본 범위는 조직 내부 의사결정용 투표이며, 법적 효력 투표는 향후 확장 모드로 둔다.

### 핵심 사용 시나리오

1. 조직 관리자가 투표를 생성하고 유권자 명부를 등록한다.
2. 승인 관리자가 투표 설정, 문항, 명부, 공개 정책을 검수하고 승인한다.
3. 유권자는 MVP 기본값인 초대 링크와 유권자 식별자 확인으로 인증한 뒤 투표한다. 1회성 인증코드는 투표별 선택 옵션이다.
4. 마감 전까지 재투표할 수 있으며 마지막 유효 Ballot만 공식 집계된다.
5. 종료 후 결과를 집계, 검수, 확정한 뒤 공개 정책에 따라 공개한다.
6. 모든 위험 작업, 결과 출력, 정정, 무효, DB 긴급 접근은 감사 로그로 남긴다.

### 기본 전제

- 서비스는 Tenant / Organization 기반 SaaS이다.
- Election별 VoterRegistry를 독립적으로 확정한다.
- 투표자 인증은 Election 또는 AuthenticationPolicy에서 선택한다. MVP 기본 인증은 "초대 링크 + 유권자 식별자 확인"이다.
- 1회성 인증코드와 SMS/카카오/외부 본인확인 등 비용 발생 인증은 기본 비활성화하며, 향후 상용화/보안 강화/법적 효력 모드에서 활성화할 수 있게 둔다.
- 익명투표에서는 EligibleVoter / VotingCredential과 Ballot / Vote를 직접 연결하지 않는다.
- 관리자 MFA, RBAC, 위험 작업 step-up 인증은 필수이다.
- 공개된 결과는 덮어쓰지 않고 ResultVersion과 정정 절차로 관리한다.

### 설계 범위

- 서비스 구조, 역할, 도메인 모델, 상태 모델
- 관리자/투표자 UX 흐름
- 데이터 모델 후보와 개인정보 보호 구조
- 익명투표 보안 모델
- 권한/보안/운영/감사 정책
- 예외 상황 대응 원칙

### 제외 범위

- API 엔드포인트 상세
- 화면 와이어프레임
- 구현 코드
- 개발 일정과 인력 계획
- 법적 효력 투표의 세부 인증/전자서명 설계

## 2. 핵심 설계 원칙

| 원칙 | 설명 |
| --- | --- |
| 투표 신뢰성 최우선 | 모든 기능은 결과를 신뢰할 수 있는지 기준으로 판단한다. |
| 폐쇄형 유권자 명부 기반 | Election별로 확정된 명부에 포함된 유권자만 투표할 수 있다. |
| 강한 익명성 | 익명투표에서 유권자 신원과 선택 내용은 저장 구조와 운영 화면 모두에서 분리한다. |
| 재투표 허용 | 마감 전까지 재투표를 허용하고 마지막 accepted Ballot만 공식 집계한다. |
| 결과 확정 후 공개 | 종료 즉시 공개하지 않고 집계, 검수, 확정 후 공개한다. |
| Published 이후 덮어쓰기 금지 | 공개된 결과는 수정하지 않고 정정 요청, 승인, 새 버전, 정정 공지로 처리한다. |
| RBAC 기반 권한 분리 | 생성, 승인, 감사, 공개, 권한 관리를 분리한다. |
| 위험 작업 감사 기록 | 중단, 종료, 무효, 정정, 권한 변경, 보고서 출력, DB 접근은 사유와 승인자를 남긴다. |
| 개인정보 최소 수집/노출 | 필요한 정보만 수집하고 암호화, 해시, 마스킹, 파기 정책을 적용한다. |
| 로그에 의한 재식별 방지 | 인증 로그와 제출 로그를 개인 단위로 연결하지 않는다. |

## 3. 사용자 역할 및 권한 개요

| 역할 | 설명 | 주요 권한 | 제한 사항 | 보안상 주의사항 |
| --- | --- | --- | --- | --- |
| Tenant | SaaS 고객 경계 | 조직, 구독, 보안 정책의 상위 단위 | 직접 업무 사용자가 아님 | 데이터 격리 기준 |
| Organization | 실제 투표 운영 단체 | 투표, 관리자, 명부 운영 범위 | 다른 조직 데이터 접근 불가 | 조직별 정책 분리 |
| SystemAdmin | SaaS 운영자 | Tenant 관리, 시스템 설정, 보안 감사 | 조직 투표 선택 내용 접근 금지 | 긴급 접근은 별도 감사 |
| OrganizationOwner | 조직 최고 관리자 | 관리자 초대, Role 부여, 조직 설정 | 결과 조작 권한 없음 | 권한 변경은 고위험 작업 |
| ElectionManager | 투표 실무 관리자 | 초안 작성, 명부 등록, 초대 관리 | 시작 승인/결과 확정 단독 불가 | 명부 개인정보 노출 최소화 |
| ElectionApprover | 승인 관리자 | 시작, 중단, 종료, 무효, 결과 확정 승인 | 초안 작성자와 분리 권장 | step-up 인증 필요 |
| Auditor | 감사자 | 설정, 로그, 보고서 열람 | 수정 권한 없음 | 로그 열람 자체도 감사 |
| ResultPublisher | 결과 공개 담당 | 확정 결과 공개, 보고서 출력 | 미확정 결과 공개 금지 | 출력 목적 기록 |
| Voter | 유권자 | 인증, 투표, 허용 결과 열람 | 관리자 기능 접근 불가 | 초대 링크와 식별 정보 공유 금지. 인증코드 사용 투표에서는 코드 공유 금지 |
| PublicViewer | 공개 열람자 | 전체 공개 결과 열람 | 제한/비공개 결과 접근 불가 | 공개 범위만 허용 |

## 4. 서비스 전체 구조

| 영역 | 목적 | 주요 기능 |
| --- | --- | --- |
| 관리자 포털 | 투표 운영과 감사를 수행하는 업무 공간 | 대시보드, 투표 생성, 명부 관리, 승인, 모니터링, 결과 확정, 보고서, 로그 |
| 투표자 포털 | 초대받은 유권자가 쉽게 투표하는 공간 | 인증, 투표 정보 확인, 제출, 완료 확인, 재투표, 결과 열람 |
| 인증/초대 영역 | 유권자 자격과 접근 권한 검증 | Invitation, VotingCredential, AuthenticationPolicy, 선택형 인증코드, 인증 이벤트, 참여 여부 |
| 투표 운영 영역 | Election 상태와 운영 예외 관리 | 생명주기, 중단/재개/종료, 장애, 무효 |
| 결과/확정 영역 | 결과 집계와 공식 결과 버전 관리 | Result, ResultVersion, 보고서, 정정, 공개 |
| 감사/로그 영역 | 운영 투명성과 분쟁 대응 | AuditEvent, SecurityEvent, CredentialEvent, SubmissionEvent, DbAccessEvent |
| 보안/개인정보 보호 영역 | 데이터와 권한 보호 | MFA, RBAC, 암호화, 해시, 마스킹, 백업, 파기 |

## 5. 도메인 모델

| 객체 | 정의 | 책임 | 주요 관계 | 설계 근거 |
| --- | --- | --- | --- | --- |
| Tenant | SaaS 고객 단위 | 데이터 격리, 구독, 보안 경계 | Organization 보유 | 멀티테넌트 운영 기준 |
| Organization | 투표 운영 조직 | 관리자와 Election 관리 | Tenant 하위 | 조직별 정책과 권한 분리 |
| User | 관리자 계정 | 로그인, MFA, Role 부여 대상 | Role과 N:M | 관리자와 투표자를 분리 |
| Role | 권한 묶음 | 업무 역할 정의 | Permission과 N:M | 작은 조직 겸임 가능 |
| Permission | 행위 권한 | 세밀한 접근 통제 | Role에 매핑 | 위험 권한 분류 가능 |
| Election | 투표 이벤트 | 일정, 상태, 정책, 문항 관리 | Question, Policy, Registry 보유 | 핵심 업무 단위 |
| ElectionPolicy | 투표 정책 | 익명/기명, 재투표, 공개, 정족수 | Election 1:1 | 정책 변경 영향 분리 |
| AuthenticationPolicy | 투표자 인증 정책 | 초대 링크, 식별자 확인, 선택형 코드, 외부 인증 등 인증 방식 정의 | Election 또는 ElectionPolicy와 연결 | MVP 기본 인증과 향후 유료/강한 인증 확장 분리 |
| State | Election 상태 | 작업 가능 범위 결정 | StateHistory로 이력화 | 시작 후 무결성 보장 |
| VoterRegistry | Election별 명부 | 유권자 범위 확정 | EligibleVoter 보유 | 시작 시점 기준 고정 |
| EligibleVoter | 투표 가능자 | 자격, 연락처, 참여 대상 | Credential, Invitation 연결 | PII 보호 필요 |
| Invitation | 초대 | 유권자에게 접근 경로 제공 | EligibleVoter 하위 | 발송/재발송 감사 |
| VotingCredential | 투표 참여 자격/인증 상태 | 투표 가능 여부, 인증 상태, 중복/재투표 통제 | EligibleVoter 연결, Ballot 연결 금지 | 1회성 코드 전용 객체가 아니며 선택 내용과 분리 |
| AnonymousVotingPass | 익명 투표권 | 투표권 발급 및 사용 상태 관리 | 인증 영역에 속하며 AnonymousBallotGroup과 영속 FK 금지 | 인증 영역은 투표권 사용 여부만 관리 |
| AnonymousBallotGroup | 익명 제출 묶음 | 랜덤 토큰 기반 Ballot 그룹화와 재투표 묶음 관리 | Ballot 보유, 유권자/인증 영역 FK 금지 | 투표 선택 영역은 선택 내용과 Ballot 이력만 관리 |
| Ballot | 제출 이력 | 한 번의 투표 제출과 접수 상태 | Vote 보유 | 기존 Ballot 수정 금지 |
| Vote | 문항별 응답 | Ballot 내 Question 응답 | Question, VoteOption | 다중 문항 지원 |
| VoteOption | 선택형 응답 | Vote와 Option 연결 | Vote/Option N:M | 복수선택 확장 |
| Result | 집계 작업 결과 | 집계값 생성 | ResultVersion 참조 | 확정 전 결과 분리 |
| ResultVersion | 공식 결과 버전 | 확정, 공개, 정정 버전 | Report 기반 | Published 덮어쓰기 금지 |
| Report | 산출물 | PDF, CSV/Excel, 감사 요약 | ResultVersion 기반 | 출력 추적 가능 |
| Dispute | 이의제기 | 접수, 검토, 처리 | Election 연결 | 결과 분쟁 대응 |
| CorrectionRequest | 정정 요청 | 공개 결과 정정 절차 | ResultVersion 연결 | 조용한 수정 금지 |
| InvalidationRecord | 무효 처리 | 무효 사유와 승인 기록 | Election 연결 | 중대 오류 대응 |
| OperationIncident | 장애/운영 사고 | 영향 범위와 조치 기록 | Election 연결 | 장애와 공정성 관리 |
| AuditEvent | 운영 감사 | 관리자 작업, 위험 작업, 출력 기록 | 주요 객체 참조 | append-only 성격 |
| SecurityEvent | 보안 감사 | 로그인, MFA, 이상 행위 | User/Credential 참조 가능 | 보안 탐지 |
| CredentialEvent | 인증 이벤트 | 초대 확인, 식별자 확인, 인증 성공/실패, 선택형 코드 발송/재발송, 외부 인증 결과 | Credential 연결 | Ballot ID 저장 금지 |
| SubmissionEvent | 제출 이벤트 | 접수, 실패, 불명확 상태 | Ballot 영역 | Credential ID 저장 금지 |
| DbAccessEvent | DB 접근 감사 | 긴급 접근과 조회 기록 | 운영자 중심 | 내부자 통제 |

## 6. 투표 생명주기

| 상태 | 의미 | 허용 작업 | 금지 작업 | 다음 상태 | 감사 |
| --- | --- | --- | --- | --- | --- |
| Draft | 초안 | 정보/문항/명부 수정, 삭제 | 초대, 투표, 집계 | ReadyForReview | 변경 이력 |
| ReadyForReview | 검수 대기 | 검토, 반려, 승인 | 투표 시작 | Draft, Approved | 검수/반려 |
| Approved | 승인 완료 | 일정 확인, 제한 수정 | 참여, 집계 | Scheduled, Notice | 승인 |
| Scheduled | 시작 예약 | 초대 발송, 공지 | 투표 참여 | Notice, Open | 발송 |
| Notice | 사전공지 | 공지, 재발송 | 투표 참여 | Open | 공지 |
| Open | 진행 중 | 투표, 재투표, 모니터링, 중단/종료 요청 | 문항/명부 수정 | Paused, Closed | 참여/중단 |
| Paused | 일시중단 | 재개, 종료, 무효 검토 | 투표 참여, 설정 수정 | Open, Closed, Invalidated | 중단/재개 |
| Closed | 종료 | 집계 시작 | 투표, 공개 | Tallying, Invalidated | 종료 |
| Tallying | 집계 중 | 집계/재집계 | 공개 | PendingConfirmation | 집계 |
| PendingConfirmation | 확정 대기 | 검수, 확정, 이의 처리 | 공개 | Confirmed, Tallying | 확정 요청 |
| Confirmed | 확정 | 공개 준비, 보고서 | 결과 덮어쓰기 | Published, Archived | 확정 |
| Published | 공개 | 열람, 출력, 정정 요청, 무효 요청 | 조용한 수정/삭제, 기존 공개 결과 덮어쓰기 | Archived, Invalidated | 공개/정정/무효 |
| Archived | 보관 | 감사 열람, 파기 검토 | 상태 되돌림 | 없음 | 보관/파기 |
| Invalidated | 무효 | 무효 사유 열람, 무효 보고서 | 결과 공개 | 없음 | 무효 |

## 7. UX 설계 요약

### 관리자 UX

| 흐름 | UX 목표 | 실수 방지 장치 | 핵심 안내 문구 |

## 8. 데이터 모델 설계 요약

### 테이블 그룹

- SaaS/RBAC: `tenants`, `organizations`, `users`, `user_mfa_methods`, `roles`, `permissions`, `role_permissions`, `user_roles`
- Election 설정: `elections`, `election_policies`, `election_state_history`, `election_change_history`, `questions`, `options`
- 유권자/인증: `voter_registries`, `eligible_voters`, `invitations`, `voting_credentials`, `authentication_policies`, `credential_events`, `anonymous_voting_passes`
- 익명 투표: `anonymous_ballot_groups`, `ballots`, `votes`, `vote_options`, `submission_events`
- 결과/보고: `results`, `result_items`, `result_versions`, `reports`, `report_exports`
- 운영 예외: `disputes`, `correction_requests`, `invalidation_records`, `operation_incidents`
- 감사/보안: `audit_events`, `security_events`, `db_access_events`

### 핵심 관계

```text
Tenant 1:N Organization
Organization 1:N Election
Election 1:1 ElectionPolicy
Election 1:1 AuthenticationPolicy
Election 1:N Question
Question 1:N Option
Election 1:1 VoterRegistry
VoterRegistry 1:N EligibleVoter
EligibleVoter 1:N Invitation / VotingCredential
Election 1:N AnonymousBallotGroup
AnonymousBallotGroup 1:N Ballot
Ballot 1:N Vote
Vote N:M Option
Election 1:N Result / ResultVersion / Report
```

### 핵심 데이터 규칙

- `ballots`와 `votes`에는 `user_id`, `eligible_voter_id`, `voting_credential_id`를 두지 않는다.
- `anonymous_ballot_groups`에도 `eligible_voter_id`, `voting_credential_id`를 두지 않는다.
- `ballot_group_token_hash`는 유권자 식별자 기반 해시가 아니라 랜덤 토큰 기반이어야 하며, 유권자 식별자에서 재계산 가능한 값이면 안 된다.
- `AnonymousVotingPass`와 `AnonymousBallotGroup` 사이에는 영속적인 직접 FK를 두지 않는다.
- `VotingCredential`은 넓은 의미의 투표 참여 자격/인증 상태를 관리하며, 1회성 인증코드를 항상 의미하지 않는다.
- 인증코드 관련 이력은 인증코드 방식이 활성화된 경우에만 생성된다.
- `CredentialEvent`는 코드 발송뿐 아니라 초대 확인, 유권자 식별자 확인, 인증 성공/실패, 외부 인증 결과 등 다양한 인증 이벤트를 표현할 수 있어야 한다.
- `Invitation`과 `VotingCredential`의 관계는 인증코드 필수 전제로 고정하지 않는다.
- `Ballot`/`Vote`에서 `EligibleVoter`, `VotingCredential`, `AnonymousVotingPass`로 직접 조인 가능한 구조를 만들지 않는다.
- 재투표 시 새 Ballot을 생성하고 마지막 accepted Ballot만 `is_current = true`로 둔다.
- 공식 집계 조건은 `is_current = true`, `acceptance_status = accepted`, `server_received_at <= election.ends_at`이다.
- Result와 ResultVersion을 분리해 공개 이후 정정 이력을 보존한다.

### 개인정보 보호 필드

| 대상 | 처리 |
| --- | --- |
| 이름, 이메일, 휴대폰, 외부 식별자 | 필드 암호화, 검색용 HMAC, 화면 마스킹 |
| 초대 토큰 | 원문 미저장, 해시만 저장 |
| 인증코드 | 인증코드 옵션 사용 시에만 생성, 원문 미저장, 해시만 저장 |
| ballot group token | 원문 미저장, 토큰 해시만 저장 |
| receipt | 해시 저장, 일부만 표시 |
| IP | 원문 지양, 마스킹/부분 해시 |
| User-Agent | 상세 원문 지양, 요약 저장 |
| 자유 의견 | 암호화, 로그 복사 금지 |

## 9. 익명투표 및 개인정보 보호 설계

익명투표의 최종 구조는 다음과 같다.

```text
[유권자/인증 영역]
EligibleVoter
VotingCredential
Invitation
CredentialEvent
AnonymousVotingPass

[익명 투표 영역]
AnonymousBallotGroup
Ballot
Vote
VoteOption
SubmissionEvent
```

- 두 영역 사이에는 Ballot/Vote 기준 직접 FK를 두지 않는다.
- AnonymousVotingPass는 투표권 발급과 사용 상태를 관리하지만 AnonymousBallotGroup, Ballot, Vote와 영속적인 직접 FK를 갖지 않는다.
- AnonymousBallotGroup은 익명 투표 제출 묶음을 관리하며, 재투표 처리를 위한 그룹 식별자는 유권자 식별자 기반 해시가 아니라 랜덤 토큰 기반이어야 한다.
- 인증 영역은 투표권 사용 여부만 관리하고, 투표 선택 영역은 선택 내용과 Ballot 이력만 관리한다.
- 구현 시 토큰 발급/사용 과정에서 일시적 매핑이 필요하더라도 해당 매핑은 최소화하고 장기 보관하지 않는다.
- 인증 방식은 Election 또는 AuthenticationPolicy에서 선택한다. MVP 기본값은 "초대 링크 + 유권자 식별자 확인"이며, 1회성 이메일/SMS 코드, 카카오/문자 알림 인증, 외부 본인확인, SSO, 법적 효력 모드용 강한 인증은 선택형 또는 향후 확장 방식이다.
- 투표자는 투표 완료 여부와 마지막 제출 시각, receipt 일부를 확인할 수 있으나 선택 내용은 재표시하지 않는다.
- 관리자는 유권자별 참여 여부는 확인할 수 있다.
- 단, 관리자는 유권자별 Ballot ID, Vote ID, AnonymousBallotGroup ID, ballot_group_token_hash를 볼 수 없다.
- 관리자는 특정 유권자의 선택 내용을 볼 수 없다.
- 관리자는 특정 유권자의 개인별 마지막 제출 시각을 기본적으로 볼 수 없다.
- 관리자 화면에서 인증 시각과 제출 시각을 개인 단위로 연결해 보여주면 안 된다.
- 참여 현황은 가능한 한 집계 단위로 제공한다.
- 필요한 경우 유권자별 상태는 "미참여 / 참여 완료 / 인증 문제 있음 / 초대 발송 실패" 정도로 제한한다.
- 재식별 위험이 있는 세부 시각, IP, User-Agent, 제출 이벤트 상세는 일반 관리자에게 노출하지 않는다.
- CredentialEvent에는 Ballot ID나 group ID를 저장하지 않는다.
- SubmissionEvent에는 EligibleVoter ID나 VotingCredential ID를 저장하지 않는다.
- IP/User-Agent는 상관분석을 줄이기 위해 마스킹 또는 요약 저장한다.
- 매우 작은 익명투표는 득표수 공개만으로도 재식별 위험이 있으므로 공개 제한 기준이 필요하다.

남는 한계는 최고 권한 내부자가 DB, 로그, 백업, 키를 모두 장악하는 경우와 인증 시각/제출 시각의 통계적 상관분석이다. Phase 4 정책에 따라 키 분리, 로그 접근 제한, JIT DB 접근, 감사 로그 무결성으로 완화한다.

## 10. 보안 설계

| 영역 | 설계 정책 |
| --- | --- |
| 관리자 인증/MFA | MFA 필수, 위험 작업 step-up, 세션 만료, 이상 로그인 알림 |
| 투표자 인증 | MVP 기본값은 초대 링크 + 유권자 식별자 확인. 1회성 인증코드와 비용 발생 인증은 투표별 선택 옵션 |
| AuthenticationPolicy | 초대 링크만 사용, 초대 링크 + 유권자 식별자 확인, 1회성 이메일/SMS 코드, 카카오/문자 알림 인증, 외부 본인확인/SSO, 법적 효력 모드용 강한 인증을 표현 |
| 초대/인증코드 | 초대 토큰 원문 저장 금지. 인증코드는 옵션 사용 시에만 해시 저장, 만료, 재발급 시 기존 코드 무효화 적용 |
| 세션 관리 | 관리자/투표자 세션 분리, 관리자 고위험 작업 재인증 |
| RBAC | Role과 Permission 분리, 작은 조직 역할 겸임 허용, 위험 기록 필수 |
| 위험 권한 | 시작/중단/종료/무효/확정/정정/권한 변경/DB 접근은 고위험 |
| 로그 및 감사 | AuditEvent append-only, 민감정보 원문 저장 금지, 익명투표에서 인증/제출 이벤트 개인 단위 연결 표시 금지 |
| DB 운영자 접근 | 원칙적 직접 접근 금지, 긴급 JIT 접근, 사유와 사후 감사 |
| 백업/복구 | 백업 암호화, 접근 승인, 복구 테스트, 파기 반영 절차 |
| 결과/보고서 | 확정 전 공개 금지, Published 이후 기존 공개 결과 보존, 새 ResultVersion과 정정/철회/무효 공지로 처리 |

AuthenticationPolicy는 각 인증 방식에 대해 기본 활성 여부, 비용 발생 여부, 보안 수준, 투표별 사용 가능 여부, 조직별 허용 여부, 향후 확장 가능 여부를 표현해야 한다. MVP에서 기본 활성화되는 방식은 "초대 링크 + 유권자 식별자 확인"이며, 초대 링크만 사용하는 방식은 낮은 보안 수준의 제한 옵션으로 다룬다. 1회성 이메일/SMS 코드, 카카오/문자 알림 인증, 외부 본인확인, SSO, 법적 효력 모드용 강한 인증은 기본 비활성 옵션으로 둔다.

## 12. 예외 상황 및 운영 대응

| 상황 | 사용자 안내 | 관리자 조치 | 허용/금지 | 감사 로그 | 데이터 고려 |
| --- | --- | --- | --- | --- | --- |
| 인증 실패 | "정보가 일치하지 않습니다." | 실패 증가 모니터링 | 유권자 식별 재시도 허용, 존재 여부 노출 금지. 코드 인증 사용 시 재발송 가능 | 인증 실패 | 실패 횟수, 잠금 |
| 코드 분실 | 코드 인증 사용 투표에서만 재발송 안내 | 재발송 제한 확인 | 제한 내 재발송 | 재발송 | 코드 옵션 사용 시 발송 이력 |
| 이미 투표 | 완료 여부와 마지막 제출 시각 | 참여 여부 확인 | 재투표 허용, 선택 재표시 금지 | 완료 조회 | 참여 상태 분리 |
| 재투표 | 마지막 제출만 유효 안내 | 재투표 수 통계 | 새 Ballot 생성, 기존 수정 금지 | 재제출 | is_current |
| 마감 직전 | 서버 접수 기준 안내 | 마감 판정 확인 | 접수 완료 시각 기준 | 제출 성공/실패 | server_received_at |
| 제출 중 오류 | 제출 여부 확인 안내 | 장애 추이 확인 | 상태 조회/재시도 | 제출 불명확 | acceptance_status |
| 완료 불명확 | receipt 일부 안내 | 완료 여부만 확인 | 선택 내용 확인 금지 | 완료 조회 | receipt_hash |
| 명부 오류 | 필요 시 공지 | 시작 전 수정, 시작 후 영향 분석 | Open 이후 직접 수정 금지 | 오류/조치 | 검증 이력 |
| 유권자 누락 | 원칙과 조치 안내 | 중단/무효 검토 | 조용한 추가 금지 | 누락 판단 | 신고/결정 |
| 투표 중 장애 | 장애 안내 | Incident 등록 | 중단/재개/연장 검토 | 장애/조치 | 영향 범위 |
| 일시중단 | 중단 사유 안내 | 승인 후 중단 | 투표 참여 금지 | 중단 | 상태 이력 |
| 무효 처리 | 무효 공지 | 사유/승인/보고 | 결과 공개 금지 | 무효 | InvalidationRecord |
| 결과 이의 | 접수/처리 안내 | 확정 보류 검토 | 미처리 무시 금지 | 이의 처리 | Dispute |
| Published 정정 | 정정/철회/무효 공지 | 기존 공개 결과 보존, 새 ResultVersion 또는 무효 기록 생성 | 덮어쓰기/조용한 삭제 금지 | 정정/무효 승인 | ResultVersion, InvalidationRecord |
| 관리자 의심 활동 | 보통 미공지 | 계정 잠금, 세션 폐기 | 권한 유지 금지 | SecurityEvent | 보안 이력 |
| DB 긴급 접근 | 보통 미공지 | JIT 승인/사후감사 | 무단 직접 접근 금지 | DbAccessEvent | 접근 영역/사유 |

Published 이후 오류가 발견되면 기존 공개 결과는 이력으로 보존한다. 정정이 필요한 경우 새 ResultVersion을 생성하고 정정 공지를 함께 남기며, 중대한 오류인 경우 Invalidated 상태 또는 무효 처리 기록을 남긴다. 무효 처리 시에도 기존 공개 결과의 존재, 무효 사유, 승인자, 승인 시각, 공지 내용을 감사 가능하게 보존한다. 공개 결과는 덮어쓰기 방식이 아니라 정정/철회/무효 공지 추가 방식으로 처리한다.
